为什么你应该用 Markdown 写文档？

我已经不记得这是第几次向人安利 Markdown 了。

就在这个博客里就发过一篇《用Markdown写日志》。

但基于以下的理由，我还是决定重新写一篇：

  1. 当初写的重点在HOW-TO，理由部分写得一塌糊涂；而这次我不打算讨论语法，重点谈为什么。
  2. 当初重点在写日志；而这次想讨论用Markdown串起所有文档的可能性。
  3. 四年过去了，整个环境有了一些变化，像Typora、nodePPT 和 Marp 这些工具的出现，让Markdown有了更多的可能，更加易用。Typora甚至到了让我写字时有幸福感的高度。

这份Markdown文件同时输出 这篇博客 和 一份演示文稿，有些地方为了迁就演示，有些排版和语气会不像博客。

一、为什么要写文档？

一个问题

==成熟的企业== 跟 初创企业 ，差别在哪

  • 设备？
  • 钱？
  • 人？
  • …

那一个 ==领域专家== 跟 一个新手，差别在哪？

知识资产（Knowledge assets）

是指企业拥有或控制的、不具有独立实物形态、对生产和服务长期发挥作用并能带来经济效益的知识。

又叫智力资产。这个解释来自MBA智库。因为MBA关注商业管理，这个是相对狭义的定义，其实个人，还有社会，都有他的知识资产。

知识资产（智力资产）

  • 人力资产
  • 市场资产
  • 知识产权资产
  • 基础结构资产
    • 管理哲学 、 企业文化
    • IT系统 、 财务结构
    • …

除了人力资产以人员、市场资产主要以品牌美誉度 和 企业关系 的方式存在，剩下的知识资产 都以某种形式的资料，主要是文件、文档的方式存在、传播 和 传承。

PMP（Project Management Professional）讲 『组织过程资产』，就是讲 项目组织在实践中形成自己独特的过程资产，构成组织的核心竞争力。

一些烦恼

  • 记录：如何做到随时随地记录？
  • 编辑：缺乏好用的编辑工具；思路总是被打断。
  • 发布：不同场景、不同用途需要不同的格式，明明差不多的内容，却整天要返工。
  • 演示：做PPT / KeyNote 吃力不讨好。
  • 版本追踪：
    • 非程序员：『版本追踪』是什么？
    • 程序员：文档还要做版本追踪？/ 我们已经做了。

首先，记录，为了保持记录的完整性有效性，以及减少事后整理的负担，应该尽可能及时地随时随地地记录。这涉及到工具的便利性，跨平台，甚至覆盖移动端，在手机上也有，还有打开速度等，尽量减少使用负担。

而事实上，很多人嫌记录麻烦，记录文档变成了一种很痛苦的例行公事，只有在任务要求时，事后回忆甚至编造一些记录应付了事。譬如说月度总结，譬如说没有commit message，事后补上。

编辑里面好的工具很重要，但还有一个，就是内容和格式同时编辑，也会引起思路的打断。

演示上面，一个是已有的文档不能简单地转换成演示文稿；另一个是，容易在一些花俏的细节上（例如动画）花过多精力，但最终的效果也不好。

版本管理 程序员一般是知道的。目前我们多数用SVN，后面有机会跟大家分享一下Git的使用。我们现在的产品文档是已经用SVN管理起来，但严格来说，没有得到很好的管理，因为你没办法像代码一样diff两个版本。因为多数文档是二进制文件，你必须两个版本都获取下来，分别打开用肉眼判断，这是很低效而且容易犯错的。

说完高大上的概念，说说我个人的体会。

我高中的时候，曾经问过为什么要写注释和文档的蠢话。那是因为小时候写的东西都太简单了。
而工作这几年，我发现无论当时多么刻骨铭心的坑，通过多个宵，你最终都会忘记，只有写下来的才是你的、你的团队的。
我觉得最可惜的事情，就是因为三星的安保政策，有很多东西我没有记下来。所以一定要有方便的工具，让你保持随时记录的习惯。

  • plain text (.txt)

  • Initialization file (.ini), Properties, YAML, JavaScript Object Notation (.json)

  • eXtensive Markup Language (.xml), HyperText Markup Language (.html)

  • Markdown (.md), reStructuredText (.rst)

当你在那个项目当中时，你可能会觉得Field19我知道是什么，那个数字我也能理解。但是上线一年后再回来维护呢？一个新来的同事去看呢？

缺点

纯文本有两个主要的缺点：

  1. 跟压缩的二进制格式相比，纯文本需要占据更多空间
  2. 解析处理纯文本，可能需要更多的运算

在某些场景下，这两个缺点是无法容忍的，譬如说数据库储存的关系数据。但随着存储空间和运算能力变得越来越廉价，这点缺点跟以下优点比起来可能是微不足道的。

文本的威力

保证不过时

人能够阅读的数据形式，以及自述的数据，将比所有其他数据形式和创建它们的应用都活得更长久。

只要数据还存在，你就有可能使用它。哪怕是创建它的程序不在的时候：有可能是这个程序过时已久；也有可能这个程序刚好在这台电脑，或者你的手机上没有。

杠杆作用

提供锋利的『小工具』，各自专注做好一件事。

当系统崩溃时，你可能需要最小环境恢复。

  • shell 工具 (diff, awk, sed, grep …)
  • 最大公约数
  • 版本管理

第一句话是 Unix的设计哲学，这个哲学基于一个公共的底层格式：面向行的纯文本文件 实现。每个工具专注做好一件事，然后通过 管道(pipe) 和 纯文本文件 互相打通，变成无所不能的shell脚本。

第二句话是说，假如系统崩溃，有些功能可能用不了，譬如图形界面。（其实不用系统崩溃，通过shell远程连接就已经是了）这个时候，纯文本的优点就出来了。

在不同的系统、不同的环境、基于网络的异构通信、自行协商的数据交换。如果需要确保各方能用一个公共标准通信，纯文本就是标准（所以我们看到XML，json的流行）

最后基于纯文本的版本管理，你可以追踪每个版本的修改，这才是真正的版本管理。

所谓的杠杆作用是指，当一个标准可以串联起一系列工具，那么它就成为了一个支点，允许你用比较小的力气，完成复杂的工作。

2.2 内容 与 样式 的分离

两顶帽子

有两顶帽子，但同时只戴其中一顶：

在修改内容时，戴上作者的帽子，不要理会显示的样式；

当内容完成，修改样式时，戴上读者的帽子，不要改动内容。

两顶帽子的原则来自面向对象编程。有接口的两顶帽子、重构的两顶帽子，等等。

强调的是每次关注改变的其中一个方面，隔离互相影响的变化。同时改变多个方面容易犯错，或者打乱思路。

受Word等软件的影响，大家似乎已经把 内容编排 和 样式 混在一起了
我举个例子：
这一段的标题是一个二级标题，上面这段话，是一段引用；这是内容编排。

具体二级标题、引用文字 应该怎么显示，这是样式。

这是不同的东西：内容在编辑完之后，就相对固定下来了；而样式会因应不同的场景，不同的用途，不同的读者，有不同的需求。

像我现在做演示文稿是一种需要，同样的内容发布到博客是一种需要。演示需要的字号肯定大很多。

甚至我换一个演示环境，投影仪的对比度比显示器低，我又要改动对比度。

他们变化的频率是不一样的，应该隔离开来。样式往往到最后根据场景来定。

Write Once, Export Everywhere

Java 当年宣传，号称Write Once, Run everywhere。今天借用一下。

Markdown不仅作为纯文本，在最坏的情况下都能被记事本打开。而且由于它有大量的工具支持，可以转换为各种场景下的文档。

  • pandoc(Typora, Smark)： HTML, Word, PDF, LaTeX, Epub…
  • Marp ： PDF 演示文稿
  • nodeppt ： HTML5 演示文稿 （特效比Marp炫）
  • hexo ： 静态博客

pandoc是一个命令行工具，你当然可以敲命令使用；不过方便起见，我会推荐使用一个前端工具，首推Typora。

2.3 为什么是Markdown？

再问一次这个问题。

因为我上面介绍的，最多是解释了为什么用纯文本。还有一般的纯文本功能太弱，所以我们需要标记语言。

但是没有解释为什么在众多标记语言里，非得是Markdown？

特性 Markdown rst HTML
标签 约十几种 约十几种 HTML4.1 共89个标签
标签读写 不解析也能理解读写 大部分不解析也能理解 需要层次嵌套和严谨闭合，复杂冗余
表达能力 仅覆盖常用格式，较弱；但可以直接写HTML补充，并有较多第三方增强（Mathjx，代码高亮） 支持格式比标准Markdown稍多 最强，整个互联网基本都是构建在HTML之上，前两者多数情况也是转换成HTML
工具支持 Github等代码托管，各大云笔记 不明 多数工具以HTML实现，但不提供直接编辑
格式支持 通过pandoc，能直接或间接导出几乎所有文档格式，且有较多编辑器前端可选 通过pandoc，能直接或间接导出几乎所有文档格式 能转成word、PDF等格式，但不能替换样式
总结 简单，但可拓展性强，支持广泛，工具多且开发活跃。 比Markdown稍复杂和强大，但热度不如Markdown，工具少很多。 严谨精确，适合浏览器解析，不方便一般人读写。
